# Git Commit 提交规范

## 概述

为了保持项目提交历史的整洁和可读性，我们采用行业标准的 Angular 规范来提交 Git commit。这种规范有助于自动生成变更日志，以及更好地追踪代码变更历史。

## 提交格式

```
<类型>: <简短描述>（不超过50个字符）

[可选的正文内容，解释更改的原因和影响]
```

## 类型说明

| 关键词 | 用途 | 示例 |
|--------|------|------|
| feat | 新功能 | feat: 新增支付模块 |
| fix | 修复 Bug | fix: 修复表单验证失效 |
| docs | 文档 | docs: 补充 API 文档 |
| style | 格式（不影响代码逻辑） | style: 调整代码缩进 |
| refactor | 重构 | refactor: 简化核心逻辑 |
| test | 测试 | test: 完善测试覆盖率 |
| chore | 构建/工具/杂项 | chore: 更新 webpack 配置 |
| perf | 性能优化 | perf: 提升接口响应速度 |
| build | 构建系统 | build: 打包配置修改 |
| ci | CI/CD | ci: 修复自动化部署脚本 |

## 详细说明

- **feat**: 新增功能特性
- **fix**: 修复 bug 或问题
- **docs**: 更新文档、注释等，不影响代码逻辑
- **style**: 代码格式调整，如空格、分号等，不改变代码逻辑
- **refactor**: 重构代码，既不修复 bug 也不添加功能
- **test**: 添加或修改测试代码
- **chore**: 修改构建脚本、配置文件等辅助工具
- **perf**: 性能优化，提升代码运行效率
- **build**: 影响构建系统的更改，如 npm 包升级等
- **ci**: CI/CD 配置、脚本的修改

## 作用域

在类型后面可以加上作用域，用来指定提交影响的范围：

```
feat(runtime): 添加新的意图处理功能
fix(devices): 修复设备发现机制
docs(CONTRIBUTING): 更新贡献指南
```

常见作用域包括但不限于：
- core: 核心功能模块
- skills: 技能管理相关
- devices: 设备管理相关
- orchestrator: 编排器相关
- intent_engine: 意图引擎相关
- runtime: 运行时相关
- utils: 工具函数相关

## 正文内容

如果需要详细说明，可以在标题下方添加正文内容，解释为什么进行此次更改，以及其影响。例如：

```
feat: 新增用户认证功能

实现基本的用户名密码认证机制，包括登录、登出功能。
这将允许用户安全地访问他们的个人数据。

Fixes: #1234
Refs: #5678
```

## 最佳实践

- 始终使用英文提交信息，以保持一致性
- 提交标题使用祈使句（如"Add feature"而不是"Added feature"）
- 第1个单词首字母大写
- 不要在末尾添加句号
- 描述简洁明了，避免过于复杂
- 提交前确保代码经过充分测试
